# ADR-0057 — EU AI Act Certification Readiness: Gap Closure

**Status:** Accepted — M1–M10 implemented 2026-05-26  
**Date:** 2026-05-26  
**Authors:** Architectural design — Forge persona (Claude Code, maintainer session)  
**Type:** Multi-component — Governance + Layer 39 + CLI tooling  
**Depends on:** ADR-0056 (compliance manifest), ADR-0042 (L34), ADR-0043 (L35),
L16 (audit chain), L19 (disclosure), L36 (erasure), L37 (audit-at-rest)  
**Related:** `docs/claude-ref/layer-security.md`, `business_model.md`,
`docs/decisions/ADR-001-opencode-eu-ai-act.md`, `docs/decisions/0041-GAP-ANALYSIS.md`

---

## Context

A compliance analysis conducted in May 2026 mapped AtelierOS's implementation
against EU AI Act 2026 requirements. The structural foundation (L10, L16, L19,
L34, L35, L36, L37, L38) is sound: disclosure, consent, audit chain,
data-flow guards, and erasure are all deterministically enforced and wired.

The analysis also confirmed that **four architectural gaps remain** which would
block a formal self-assessment certification or third-party audit:

| Gap | Regulation | Architectural? |
|---|---|---|
| No machine-readable content marking format | Art. 50 §4 | Yes — format unspecified in code |
| No incident tracking mechanism | Art. 73 | Yes — audit chain exists but no incident SOP or structure |
| No operator obligation acknowledgment gate | Art. 28–30 | Yes — templates exist, no boot-time enforcement |
| ADR-0056 M1–M6 all Draft | ADR-0056 | Yes — Compliance Manifest has no implementation |

Two additional gaps are **documentation artifacts**, not architectural deficiencies.
They can be generated from existing compliance data rather than built from scratch:

| Gap | Path |
|---|---|
| Annex IV Technical Documentation (Art. 43) | Generated by CLI from manifests + ADRs |
| Risk Classification Statement | `docs/compliance/RISK-CLASSIFICATION.md` (static doc) |

The EU AI Act entered full applicability on **2026-08-02**. AtelierOS is
positioned on EU compliance as a primary market differentiator (see
`business_model.md`). A credible self-assessment package — covering the
four architectural components and two documentation artifacts — is the
minimum bar for a Notified Body review or BSI-Grundschutz certification
engagement. That engagement has an 18–36 month lead time, meaning the gap
closure must begin no later than Q3 2026.

**Scope of this ADR:**

This ADR introduces the four missing architectural components, specifies
the two documentation artifacts that derive from them, and defines a
certification package CLI that bundles everything a certifier needs into
a single exportable directory.

What this ADR explicitly does **not** cover:
- BSI-Grundschutz certification process itself (operator responsibility).
- DPO/Legal sign-off on documentation artifacts (external actors).
- High-risk classification (AtelierOS is Limited Risk — see § Risk
  Classification below; Annex III workflows require additional operator action).
- Any changes to existing structural layers (L10–L38) — those are correct
  and remain untouched.

## Risk Classification (Authoritative)

**AtelierOS standalone classification: Limited-Risk AI System**

AtelierOS is an orchestration layer over external LLMs. It does not make
automated decisions with legal or similarly significant effect. It does not
perform biometric categorisation, critical infrastructure management, or any
other Annex III function.

- It does **not** train, fine-tune, or serve its own model weights.
- It does **not** autonomously take consequential actions outside of a
  human-in-the-loop chat session.
- Its disclosure and consent architecture is structurally enforced
  (L19 + L16 Phase 4), not policy-document-only.

Per EU AI Act Annex I (Article 3(1)): AtelierOS is a software system
exhibiting the behaviour of an AI system by interacting with users in natural
language. It is **not** a General-Purpose AI Model (no weights published).
It is **not** a High-Risk AI System as defined by Annex III.

**Conditional reclassification:** If a tenant operator deploys AtelierOS as a
component in an Annex III workflow (e.g., CV screening, credit-eligibility
scoring, benefit-assessment), the operator bears the Art. 25-28 cascade
obligations. AtelierOS provides the structural audit, consent, and data-flow
machinery required by those obligations (L16, L34, L35, L36, L37), but it
does not substitute for the operator's own AI system registration, conformity
assessment, and Notified Body engagement for that deployment.

This classification must be reviewed with every MAJOR Annex update by the
European Commission. It is recorded in `docs/compliance/RISK-CLASSIFICATION.md`
(generated by `atelier-annex-iv generate --section risk-classification`).

## Decision Drivers

1. **August 2026 deadline** — Full EU AI Act applicability means every
   disclosure, audit, and oversight requirement is now legally binding.
2. **Certification as moat** — A verified self-assessment is the single
   hardest thing for competitors to copy quickly. The gap closure enables it.
3. **Art. 73 is structural, not documentary** — Serious incident tracking must
   be anchored in the audit chain and follow a defined schema. A Word-document
   incident log is not auditable.
4. **Art. 50 §4 is underspecified in the codebase** — "provenance metadata in
   adapter envelope" is the invariant, but the format is undefined. A certifier
   will ask for the machine-readable spec; there is none.
5. **Operator obligation acknowledgment must be enforceable** — DPIA and
   privacy-notice templates exist, but nothing prevents a tenant from going to
   production without completing them. A boot-time gate fixes this.
6. **ADR-0056 M1–M6 are load-bearing for the entire governance architecture**
   — The Compliance Manifest is the machine-readable source of truth for CI
   enforcement and future audit. Until M1–M6 are implemented, it is a document,
   not a system.
7. **Annex IV generation from existing data** — AtelierOS already produces all
   the inputs required for Annex IV (manifests, ADR layer mapping, test refs,
   audit chain stats, retention config). A generator CLI makes the document
   reproducible and updateable, not a one-off PDF artifact.

## Decision

### Component 1 — Content Marking Specification (Art. 50 §4)

**Status:** Requires specification + adapter update.

Art. 50 §4 requires AI-generated content published to natural persons to carry
a machine-readable mark. The current invariant ("every outbound message carries
provenance metadata in adapter envelope") is structurally present but the format
is nowhere specified or tested.

#### Envelope Extension

Each outbound message from any bridge adapter must carry a `provenance` block
in its internal envelope before delivery to the bridge transport:

```json
{
  "message_id": "msg_01J...",
  "content": "...",
  "provenance": {
    "ai_generated": true,
    "generator_id": "claude_code",
    "model_hint": "claude-sonnet-4-6",
    "persona": "assistant",
    "atelier_instance": "<instance_uuid>",
    "session_id": "discord:1234567890__1234567890",
    "timestamp_utc": "2026-08-01T14:22:00Z",
    "manifest_version": "1.0.0"
  }
}
```

Field semantics:

| Field | Required | Source |
|---|---|---|
| `ai_generated` | Always `true` | Hardcoded — every AtelierOS message is AI-generated |
| `generator_id` | Yes | Engine name from `WorkerEngine.name` |
| `model_hint` | Best-effort | Parsed from engine response headers or config |
| `persona` | Yes | Active persona name at spawn time |
| `atelier_instance` | Yes | L38 `instance_id.json` UUID |
| `session_id` | Yes | `<bridge>:<chat_key>` |
| `timestamp_utc` | Yes | UTC at envelope construction |
| `manifest_version` | Yes | Active compliance manifest version |

`model_hint` is best-effort. If the engine does not surface the model name
(e.g., Ollama with opaque endpoint), the field is set to `"unknown"`. The
certifier requirement is machine-readability, not perfect model attribution.

#### Bridge Transport Mapping

Each bridge adapter maps `provenance` to its transport's native mechanism:

| Bridge | Transport | Mapping |
|---|---|---|
| Discord | REST + embed | `provenance` stored in `message.embeds[0].footer.text` as base64-encoded JSON; also present in in-memory envelope |
| WhatsApp | CloudAPI | `provenance` serialised as a `referral` context field |
| CLI | stdout | `provenance` emitted as JSON comment line (`// provenance: {...}`) after each response |

The envelope's `provenance` block is **never** stripped by the adapter layer.
If a bridge transport has no mechanism for machine-readable metadata, the
maintainer must document the limitation and add it to `eu-ai-act.yaml` as a
known constraint for that bridge.

#### Audit Event

```json
{
  "event": "content_marking.applied",
  "generator_id": "claude_code",
  "persona": "assistant",
  "session_id": "discord:...",
  "manifest_version": "1.0.0"
}
```

Audit allow-list: `generator_id`, `persona`, `session_id`, `manifest_version`.
Never: `content`, `model weights`, `prompt`, `provenance.atelier_instance`.

**Must NOT do:**
- Strip `provenance` block in any bridge adapter.
- Let `ai_generated` be `false` or absent for any AtelierOS-generated message.
- Put message content in the audit event.
- Require `model_hint` to be exact — best-effort is sufficient.

### Component 2 — Layer 39: Incident Tracker (Art. 73)

**Status:** Requires new module.

Art. 73 requires deployers of AI systems to notify supervisory authorities
about serious incidents within 15 days of detection. "Serious incident" in
the AtelierOS context is defined as:

| Category | AtelierOS-specific definition |
|---|---|
| **Chain integrity** | `audit.chain_gap_detected` CRITICAL event |
| **Consent bypass** | Any message processed without prior `consent.granted` for that uid |
| **Engine policy violation** | `data_flow.blocked` or `egress.blocked` event where engine executed despite denial (i.e., gate failed open on a policy-explicit deny) |
| **PII in audit chain** | A regex scan of the most-recent audit segment detects unredacted email, name, or phone pattern |
| **Secret exposure** | Any `path_gate.denied` event that includes a vault path as the attempted write target, where the write succeeded (i.e., path-gate self-test failed at boot) |
| **Disclosure failure** | A user session reached `bridge.message_received` without a prior `disclosure.shown` event for that (channel, uid) pair |

These definitions are conservative: they describe failures of structural
invariants that AtelierOS already enforces. A "serious incident" in this
context is a structural control failure, not a normal operational event.

#### Module: `operator/bridges/shared/incident_tracker.py`

```python
@dataclass
class IncidentRecord:
    incident_id: str          # UUID4
    detected_at: str          # ISO-8601 UTC
    category: str             # one of the 6 categories above
    severity: str             # "serious" | "warning" | "informational"
    trigger_event: str        # audit event_type that triggered detection
    trigger_chain_hash: str   # first 16 hex chars of triggering audit entry hash
    description: str          # operator-authored free text — stored in record,
                              # NEVER in audit chain
    status: str               # "open" | "contained" | "notified" | "closed"
    notified_at: str | None   # ISO-8601 UTC of supervisory authority notification
    closed_at: str | None
```

Storage: `<tenant>/global/incidents/<incident_id>.json` (mode 0600, atomic
write + flock). The `description` field is stored only here, never in the
audit chain — the same principle as L36's `ErasureRequest.notes`.

#### Audit Events

```json
{ "event": "incident.opened",    "incident_id": "...", "category": "...",
  "severity": "serious", "trigger_event": "audit.chain_gap_detected",
  "trigger_chain_hash": "a1b2c3d4e5f60001" }

{ "event": "incident.status_changed", "incident_id": "...",
  "old_status": "open", "new_status": "notified" }

{ "event": "incident.closed",   "incident_id": "...", "duration_hours": 36 }
```

Audit allow-list: `incident_id`, `category`, `severity`, `trigger_event`,
`trigger_chain_hash` (16 hex), `old_status`, `new_status`, `duration_hours`.
Never: `description`, operator notes, DPO name, authority contact.

#### Auto-Detection Hook

`incident_tracker.py` exports an `IncidentAutoDetector` that the adapter
calls on every CRITICAL audit emit (PostAudit hook). If the emitted event
matches one of the 6 category triggers, the detector creates an `incident.opened`
record automatically. The operator is still responsible for reviewing, containing,
and notifying; auto-detection only ensures no CRITICAL event goes untracked.

```python
class IncidentAutoDetector:
    TRIGGER_EVENTS = {
        "audit.chain_gap_detected": "chain_integrity",
        "data_flow.blocked":        "engine_policy_violation",
        "egress.blocked":           "engine_policy_violation",
        "path_gate.denied":         "secret_exposure",
    }

    def on_audit_event(self, event: dict, tenant_id: str) -> None:
        category = self.TRIGGER_EVENTS.get(event.get("event"))
        if category:
            self._open_incident(category, event, tenant_id)
```

Consent bypass and disclosure failure require periodic batch detection
(not per-event) — they are detected by `atelier-incident scan` (see CLI).

#### CLI: `atelier-incident`

```bash
# Open an incident manually
atelier-incident open --category chain_integrity \
    --trigger-event audit.chain_gap_detected \
    --trigger-chain-hash a1b2c3d4e5f60001 \
    --description "Chain gap detected in segment-003.jsonl"

# List open incidents
atelier-incident list [--status open|contained|notified|closed]

# Show full incident record (including description)
atelier-incident show <incident_id>

# Update status
atelier-incident update <incident_id> --status notified

# Generate notification draft for supervisory authority (Art. 73 §2)
atelier-incident notify-draft <incident_id> \
    --authority BSI \
    --operator-name "Acme GmbH" \
    --output notification-<incident_id>.md

# Scan audit chain for latent consent / disclosure failures
atelier-incident scan --since 30d

# Export all incidents for audit package
atelier-incident export --output incidents-export.json
```

`notify-draft` produces a structured Markdown draft in the format recommended
by BSI / ENISA: incident description, timeline, impact assessment, root cause,
containment steps, and preventive controls. The operator fills in the blanks
and submits. AtelierOS does not auto-submit.

**Must NOT do:**
- Put `description` or any free-form operator text in the audit chain.
- Auto-submit to a supervisory authority (operator decision only).
- Use `incident_tracker.py` to import `anthropic` (CI AST lint enforces).
- Treat an `incident.opened` with `severity: informational` as triggering
  the Art. 73 15-day notification clock — that clock starts only on `severity: serious`.
- Close an incident without recording either `notified_at` or an explicit
  decision that the incident does not meet the Art. 73 threshold.

### Component 3 — Operator Declaration Gate (Art. 28–30)

**Status:** Requires `tenant.atelier.yaml` extension + boot check.

Art. 28–30 impose obligations on deployers: performing a DPIA, designating a
DPO (if required), understanding permitted and prohibited use cases, and
monitoring for drift. AtelierOS cannot perform these actions for the operator,
but it can require the operator to formally acknowledge them before the system
accepts production traffic.

#### Tenant Config Extension

```yaml
spec:
  operator_declaration:
    version: "1.0"           # matches docs/compliance/OPERATOR-OBLIGATIONS.md version
    declared_by: "J. Müller" # operator DPO or legal contact
    declared_at: "2026-07-01T09:00:00Z"
    permitted_use: "internal-coding-assistant"  # free text, operator-owned
    dpia_completed: true
    dpia_date: "2026-06-15"
```

The field is **optional** in `personal` and `dev` deployment profiles.
It is **required** (boot CRITICAL) in `eu_production` and `eu_production_ollama`
deployment profiles.

#### Boot Check (bridge.sh doctor)

```
✓ operator_declaration  version=1.0  declared_by=J. Müller  dpia=2026-06-15
```

If the declaration is absent or `dpia_completed: false` in an `eu_production`
profile:

```
CRITICAL  operator_declaration missing — EU AI Act Art. 28–30 compliance
          requires a completed operator declaration before production traffic.
          See docs/compliance/OPERATOR-OBLIGATIONS.md
```

Severity: CRITICAL. The adapter does not start.

#### Audit Event

```json
{ "event": "operator.declaration_verified",
  "declaration_version": "1.0",
  "dpia_completed": true,
  "dpia_date": "2026-06-15",
  "deployment_profile": "eu_production" }
```

Audit allow-list: `declaration_version`, `dpia_completed`, `dpia_date`,
`deployment_profile`. Never: `declared_by` (PII), `permitted_use` text,
DPO contact details.

**Must NOT do:**
- Require the operator declaration in non-production profiles (breaks dev workflow).
- Put `declared_by` (a person's name) in the audit chain.
- Accept a declaration where `dpia_completed: false` in `eu_production` profile.
- Auto-populate the declaration from any config template (operator must write it).

### Component 4 — Annex IV Generator CLI

**Status:** Requires new CLI tool.

Annex IV of the EU AI Act specifies 14 required elements of Technical
Documentation for AI systems subject to conformity assessment. AtelierOS
already has all inputs scattered across manifests, ADRs, test suites, and
audit stats. This component assembles them into a reproducible Annex IV
document.

#### CLI: `atelier-annex-iv`

```bash
# Generate full Annex IV document (Markdown)
atelier-annex-iv generate \
    --manifest compliance/eu-ai-act.yaml \
    --gdpr-manifest compliance/gdpr.yaml \
    --adr-dir docs/decisions/ \
    --test-results operator/bridges/run-all-tests.sh \
    --output docs/ANNEX-IV-TECHNICAL-DOCUMENTATION.md

# Generate individual section
atelier-annex-iv generate --section risk-classification
atelier-annex-iv generate --section training-data
atelier-annex-iv generate --section oversight-measures
atelier-annex-iv generate --section test-results

# Validate that output covers all 14 required Annex IV elements
atelier-annex-iv validate docs/ANNEX-IV-TECHNICAL-DOCUMENTATION.md
```

#### Annex IV Section Mapping

| Annex IV Element | Source in AtelierOS |
|---|---|
| 1. General description | `docs/overview.md` + `docs/for-organizations.md` |
| 2. Intended purpose | `spec.operator_declaration.permitted_use` (per-tenant) |
| 3. Interaction with other systems | L22 WorkerEngine, L38 A2A protocol (ADR-0048) |
| 4. Risk classification | This ADR § Risk Classification |
| 5. Training data description | Static: "No training — external LLM orchestration" |
| 6. Risk assessment | `compliance/eu-ai-act.yaml` rule severity table |
| 7. Human oversight | L18 Roles, L21 Proposals, L14 LDD-Toggle, engine-policy |
| 8. Accuracy, robustness, security | `operator/bridges/run-all-tests.sh` output + L16 audit |
| 9. Testing procedures | ADRs per layer, `docs/claude-ref/tests.md` |
| 10. Standards applied | GDPR Art. 6, 7, 17, 30, 32; EU AI Act Art. 14, 28–30, 50, 73 |
| 11. EU declaration of conformity | Generated by `atelier-annex-iv generate --section declaration` |
| 12. Post-market monitoring plan | `atelier-incident scan` schedule + L37 audit retention |
| 13. Version history | `compliance/CHANGELOG.md` |
| 14. Instructions for use | `docs/compliance/DPIA-TEMPLATE.md`, `docs/compliance/OPERATOR-OBLIGATIONS.md` |

The generator reads existing source files. It does not invent information.
Where a source file does not exist, the section is generated with a
`[OPERATOR: FILL IN]` placeholder clearly marked. A run with remaining
placeholders fails `atelier-annex-iv validate` with a non-zero exit code.

**Must NOT do:**
- Auto-generate the "intended purpose" section (operator-authored only).
- Put LLM-generated prose in Annex IV without human review marker.
- `import anthropic` from `annex_iv_generator.py` (CI AST lint enforces).
- Submit the generated document without operator sign-off.

### Component 6 — Certification Package Export

A new top-level CLI command bundles all certification artifacts into a
single directory suitable for delivery to a Notified Body, BSI evaluator,
or DPO:

```bash
atelier-annex-iv export-package \
    --output-dir /tmp/atelier-cert-package-$(date +%Y%m%d) \
    --tenant _default
```

Contents of the package:

```
atelier-cert-package-20260801/
├── README.md                          # Navigation guide for certifier
├── ANNEX-IV.md                        # Generated by atelier-annex-iv generate
├── RISK-CLASSIFICATION.md             # Static doc (this ADR § Risk Classification)
├── OPERATOR-OBLIGATIONS.md            # docs/compliance/OPERATOR-OBLIGATIONS.md
├── INCIDENT-RESPONSE-PLAN.md          # docs/compliance/INCIDENT-RESPONSE-PLAN.md
├── compliance/
│   ├── eu-ai-act.yaml                 # Manifest (signed copy)
│   ├── gdpr.yaml
│   └── manifest.sig
├── audit/
│   ├── compliance-check-report.json   # atelier-compliance-check --json output
│   └── incidents-export.json          # atelier-incident export output
├── tests/
│   └── test-summary.txt               # run-all-tests.sh last output
└── declarations/
    └── operator-declaration.yaml      # tenant spec.operator_declaration (PII-stripped)
```

The package is reproducible: given the same repo state and tenant config,
`export-package` produces byte-for-byte identical non-timestamp content.
A GPG-signed SHA-256 manifest of the package is generated alongside it.

## Alternatives Considered

### A. Delegate Annex IV entirely to operator (rejected)

Treat Annex IV as a purely external obligation — operators write it
themselves with no tool support.

**Rejected because:**
- AtelierOS's self-positioning as a compliance-ready platform requires that
  the 14 elements can be verified against the codebase, not just asserted.
- Without a generator, Annex IV becomes stale immediately after any
  layer change, since updating it manually will not happen reliably.
- The generator enables continuous compliance: a changed ADR triggers
  a CI check that verifies the generated doc is regenerated.

### B. Build an online Declaration of Conformity portal (rejected)

A web service (cloud.atelier.eu) hosts operator-specific Declarations of
Conformity, pulls manifest versions, and generates Annex IV on demand.

**Rejected because:**
- Introduces an external availability dependency for a certification-critical
  function.
- Certifiers and regulators expect offline-reproducible documentation.
- Operator data (DPIA details, permitted use) would need to be uploaded
  to the portal, introducing a data-minimisation concern under GDPR.
- Adds cloud infrastructure cost and complexity before the core gaps are closed.

Can be built later as an ergonomic enhancement, not a substitute.

### C. Encode incident categories in CLAUDE.md (rejected)

Document the six incident categories as CLAUDE.md rules rather than
building `incident_tracker.py`.

**Rejected because:**
- CLAUDE.md is prose for the assistant; it is not auditable, not
  machine-readable, and not anchored in the L16 hash chain.
- An incident that is documented but not recorded in the audit chain does not
  satisfy Art. 73's requirement for verifiable reporting history.
- The auto-detection hook (IncidentAutoDetector) can only exist as a
  proper module, not as a documentation entry.

**Must NOT do:**
- Remove the `provenance` block from any outbound message in any bridge adapter.
- Let `incident_tracker.py` or `annex_iv_generator.py` `import anthropic`
  (CI AST lint enforces).
- Put `IncidentRecord.description`, operator DPO name, or authority contact
  into the L16 audit chain.
- Accept a `eu_production` boot without a complete `spec.operator_declaration`
  (including `dpia_completed: true`).
- Auto-submit any incident notification to a supervisory authority.
- Treat `severity: informational` incidents as triggering the 15-day Art. 73
  notification clock.
- Include message content, prompt text, or any personal data in the
  `content_marking.applied` audit event.
- Ship the Annex IV document without `atelier-annex-iv validate` passing
  (no remaining `[OPERATOR: FILL IN]` placeholders).
- Allow the Certification Package to contain files with mode wider than 0640.
- Treat the generated Annex IV document as legally binding without operator
  sign-off and DPO review.
- Use this ADR to modify any existing structural layer (L10–L38).

## Open Questions

1. **L39 severity thresholds** — Should `data_flow.blocked` always auto-open
   a `severity: serious` incident, or only when the gate failed open (engine
   executed despite the denial)? The conservative reading is: a denied spawn
   is a control success, not a failure. Only a denied spawn that *executed
   anyway* is an incident. Needs a concrete test to validate gate-fail-open
   detection.

2. **Content marking on streaming responses** — The `provenance` block is
   added to the final envelope. For streaming responses, the first chunk
   arrives before the full response is known. Should `provenance` be injected
   as a preamble chunk, or only on the final `turn_completed` event?
   Preamble is more consistent with Art. 50 §4's "at the time of publication"
   wording.

3. **Multi-jurisdiction manifest** — ADR-0056 open question #2 (UK AI Safety
   Institute, US state laws). This ADR does not resolve it; it defers to
   ADR-0056's answer. But `atelier-annex-iv generate` should support a
   `--jurisdiction` flag for future expansion.

4. **Annex IV version pinning** — If the operator commits
   `docs/ANNEX-IV-TECHNICAL-DOCUMENTATION.md` and then a layer changes,
   is the committed doc stale? Proposed: add a SHA-256 hash of the generated
   doc to `compliance/CHANGELOG.md`; CI checks that the hash matches the
   current generator output on every PR touching a mapped source file.

5. **Incident scan frequency** — Consent-bypass and disclosure-failure
   detection require batch scanning of the audit chain, not per-event
   triggers. How frequently should `atelier-incident scan` run?
   Proposed: daily via `systemd` timer, same schedule as `atelier-session-timeout.timer`.
